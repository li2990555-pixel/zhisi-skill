# zhisi-skill Skill设计文档

## 一、Skill产品定位

- **Skill名称**: `zhisi-skill`
- **版本**: v1.0.0（规划中）
- **一句话价值**: 为硕博研究生提供人文学科论文写作的思维导航与结构检视，让非线性写作不迷路、不跑偏、不漏项
- **不做什么**:
  - 不做文献管理（可建议使用 Zotero 导入）
  - 不做语法润色
  - 不做查重降重
  - 不替用户写论文（辅助而非替代）
  - 不自动生成全文
- **触发模式**: 主动触发 —— 用户明确调用 Skill 的某个子功能
- **生成时间**: 2026-06-18

---

## 二、目标用户

### 2.1 用户画像

- **核心用户**: 人文社科硕博研究生（哲学、中国语言文学、艺术学、历史学等）
- **技术水平**: 非技术背景，偏好自然语言交互，需渐进式引导
- **使用场景**:
  - 论文开题后，需要搭建或优化论文结构
  - 写作过程中跳写了多个章节，需要整体检视
  - 收集了大量文献材料，不确定如何分配
  - 写完初稿后，需要检查论证漏洞和逻辑一致性
  - 写作卡壳，需要突破提示
  - 论文完成度不明，需要健康度评估

### 2.2 用户痛点

| 痛点 | 严重程度 | Skill如何解决 |
|------|----------|---------------|
| 结构逻辑难组织 | 高 | 大纲设计、逆序写作引导、非线性草稿重组 |
| 跳写导致结构失控 | 高 | 查漏补缺、写作健康度仪表盘 |
| 材料管理混乱 | 中高 | 材料库梳理、跨学科视角推荐 |
| 缺乏专业检视工具 | 高 | 论证强度分析、术语一致性检查、学术对话感知、冗余检测 |
| 写作卡壳 | 中 | 写作障碍突破、批注式写作 |

## 三、Skill架构设计

### 3.1 核心架构模式：对话引擎 + 分析引擎

```
用户交互层（主对话引擎）
  ├── 结构设计模式（A1-A3）
  ├── 写作体验模式（B1-B2）
  ├── 智能检视模式（C1-C5）
  ├── 知识管理模式（D1-D2）
  └── 全局总览模式（E1）
          │
          ▼
分析引擎（可选 subagent 模式）
  ├── 文本扫描器（用于查漏补缺、术语检查、冗余检测）
  ├── 论证分析器（用于论证强度分析、对话感知）
  ├── 重组分析器（用于非线性草稿重组）
  └── 仪表盘聚合器
```

### 3.2 路由机制：能力选择 → 模式分发

```
用户触发 Skill
  │
  ▼
能力选择（用户选或自动识别）
  ├── 【模式A】结构设计 → A1大纲设计 / A2逆序引导 / A3草稿重组
  ├── 【模式B】写作体验 → B1批注式写作 / B2写作障碍突破
  ├── 【模式C】智能检视 → C1查漏补缺 / C2论证分析 / C3术语检查 / C4冗余检测 / C5学术对话感知
  ├── 【模式D】知识管理 → D1材料库梳理 / D2跨学科推荐
  ├── 【模式E】全局总览 → E1写作健康度仪表盘
  └── 【模式F】综合检视 → 按写作阶段组合多个模式
          │
          ▼
  执行对应模式 → 输出结果 → 推荐下一步
```

### 3.3 全平台兼容设计

**核心原则**：核心逻辑全部用对话 Prompt 表达，平台特定能力作为可选增强

| 能力层级 | 所有平台通用 | Claude Code 增强（可选） |
|---------|-------------|------------------------|
| 核心交互 | 对话引导 + 自然语言处理 | AskUserQuestion 结构化交互 |
| 分析能力 | 内置分析逻辑（对话中完成） | Agent subagent 并行分析 |
| 文件操作 | 平台内置文件接口 | Read/Write/Bash 文件系统 |
| 引用验证 | 对话中提示用户自查 | WebSearch 联网验证 |
| 持久化 | 平台记忆/笔记功能 | 本地文件系统 Markdown |

## 四、对话流设计（Skill UI）

### 4.1 交互模式

- **模式**: 混合模式 —— 渐进式引导为主，但用户可在任何步骤自由跳到其他模式
- **预估交互轮次**: 3-8 轮（取决于选的模式）
- **默认路径**: 用户选择模式 → 输入内容 → Skill 分析 → 输出结果 → 推荐下一步

### 4.2 通用入口流程

```
用户主动触发 Skill
  │
  ▼
[欢迎信息] 显示 Skill 能力全景 + 当前写作状态预览
  │  ├─ 检测 docs/writings/ 中已有的大纲/草稿进度
  │  └─ 显示推荐入口（根据写作阶段）
  │
  ▼
[模式选择] 用户选择要使用的功能模块
  │  A) 结构设计 ── 大纲、逆序、重组
  │  B) 写作体验 ── 批注、障碍突破
  │  C) 智能检视 ── 查漏补缺、论证分析等
  │  D) 知识管理 ── 材料库、跨学科推荐
  │  E) 全局总览 ── 写作健康度仪表盘
  │  F) 综合检视 ── 一键全检
  │  G) 我说我的需求，你帮我判断用什么模式
  │
  ▼
[模式执行] 进入对应的子流程（3-5轮交互）
  │
  ▼
[输出] 对话输出 + 可选的 Markdown 文件持久化
  │
  ▼
[完成提示] 摘要 + 动态推荐下一步
```

### 4.3 子流程详细设计

#### 模式A1 — 大纲设计

```
① 用户输入当前提纲（可残缺）
  │
② Skill 识别论文类型：思辨型 / 历史型 / 比较型 / 实证型
  │
③ 分析缺失部分 → 给出补全建议
  │
④ 生成思维导图结构树（Markdown 格式）
  │
⑤ 输出 + 推荐下一步（如进入模式C1查漏补缺）
```

**核心 Prompt 逻辑**：
```
你是一名人文学科论文结构设计专家。请基于用户提供的提纲：
1. 识别论文的论证类型（思辨/历史/比较/实证）
2. 检测结构上缺失的关键部分（研究综述、理论框架、方法论等）
3. 对每一章的逻辑承接关系给出评价
4. 输出完整的结构建议，使用 Markdown 层级列表呈现思维导图
```

#### 模式A2 — 逆序写作引导

```
① 用户输入已写好的结论/结语章节
  │
② Skill 提取核心论点
  │
③ 反向推导：要得出此结论，导论需要铺设哪些问题？
  │
④ 输出"问题链" + 每部分对应的建议内容
```

#### 模式A3 — 非线性草稿重组

```
① 用户输入各章节已写片段（不限顺序）
  │
② Skill 分析每个片段的：论点、证据、功能定位
  │
③ 检测逻辑流 → 生成 1-3 种重组方案
  │
④ 每种方案标注：逻辑优势、代价、推荐度
```

#### 模式C2 — 论证强度分析

```
① 用户输入需要检查的章节内容
  │
② Skill 识别论证结构（前提→推理→结论）
  │
③ 逐条检测逻辑谬误：
  │  - 循环论证、偷换概念、稻草人、不当类比、以偏概全、诉诸权威
  │
④ 输出定位到具体句子的分析报告
```

**核心 Prompt 逻辑**（逻辑谬误检测表作为 reference 注入）：

检测规则（精简版）：
1. **循环论证**: 论据与结论本质上是同一命题的不同表达
2. **偷换概念**: 关键术语在推理过程中含义发生变化
3. **稻草人**: 曲解/弱化对立观点以便于反驳
4. **不当类比**: 类比的跨度超出理论合理范围
5. **以偏概全**: 从有限个案推出普遍性结论
6. **诉诸权威**: 以权威代替论证，未给出独立推理
7. **逻辑跳跃**: 前提与结论之间缺少必要的过渡环节
8. **虚假两难**: 将复杂问题简化为非此即彼的选择

#### 模式C5 — 学术对话感知

```
① 用户输入当前章节内容
  │
② Skill 提取涉及的学者、理论、流派
  │
③ 建立对话网络：用户的观点 ↔ 学者A ↔ 学者B
  │
④ 标记对话状态：
  │  🔵 继承关系 — "延续了伽达默尔的诠释学路径"
  │  🟣 批判关系 — "与福柯对话但未回应该领域的关键反驳"
  │  ⚪ 空白地带 — "此概念已有重要讨论，建议纳入对话"
  │  ❌ 伪对话 — "引用某学者但未真正与之对话"
  │
⑤ 输出学术对话网络报告
```

#### 模式E1 — 写作健康度仪表盘

```
① 用户提供各章节内容和进度概况
  │
② Skill 多维度分析：
  ├── 📐 结构完整性
  ├── 🔗 逻辑连贯性
  ├── 📚 引用覆盖率
  ├── 🎯 论证强度
  ├── 🔄 术语一致性
  └── 📝 字数进度
  │
③ 可视化仪表盘（Markdown 进度条）
  │
④ 每个维度展开细节建议
```

### 4.4 兜底机制

| 异常场景 | 处理方式 |
|----------|----------|
| 用户输入模糊 | 提供选项引导，而非拒绝。最后选项总是"其他（手动输入）" |
| 跨模式跳转 | 用户可以在任何步骤要求"切换到模式X"，自动保存当前上下文后跳转 |
| 空输入（首次使用） | 自动进入模式A1大纲设计，从零开始搭建结构 |
| 材料缓存不足 | 提示用户输入更多材料，或建议先进入模式D1材料库梳理 |
| 用户不了解什么模式适合 | 选项G — 让用户描述需求，Skill 自动判断推荐模式 |

## 五、输出与交付物

### 5.1 输出文件清单

| 文件路径 | 内容描述 | 格式 |
|----------|----------|------|
| `docs/writings/outline.md` | 当前论文的完整结构大纲（思维导图） | Markdown |
| `docs/writings/material-library.md` | 材料库梳理结果 | Markdown |
| `docs/writings/progress.md` | 查漏补缺进度表 | Markdown |
| `docs/writings/inspection-reports/论证分析报告.md` | 论证强度分析结果 | Markdown |
| `docs/writings/inspection-reports/术语一致性报告.md` | 术语检查结果 | Markdown |
| `docs/writings/inspection-reports/学术对话网络.md` | 学术对话感知分析 | Markdown |

### 5.2 质量检查清单

- [ ] 输出结果定位到具体章节/句子，而非笼统评价
- [ ] 每个建议都给出明确理由和修改方向
- [ ] 思维导图/仪表盘使用 Markdown 可视化表达
- [ ] 输出的每个判断都有文本依据支撑
- [ ] 已完成时推荐 1-3 个下一步可用的模式

### 5.3 Red Flags（不应出现的信号）

| 误区 | 正确做法 |
|------|---------|
| 输出空泛的建议（"建议加强论证"） | 定位到具体句子：第几章第几段第几句，什么逻辑问题 |
| 替用户写段落 | 只分析、只建议、只指路，不生成正文 |
| 一次性输出过多信息 | 控制单次输出长度，提供展开查看的选项 |
| 忽略用户已有的写作内容 | 每次分析都先确认和引用用户已写内容 |

## 六、跨平台适配

### 6.1 目标平台列表

| 平台 | 适配状态 | 说明 |
|------|---------|------|
| **Claude** | ✅ 完整支持 | 对话交互 + 文件持久化 + subagent 并行增强 |
| **ChatGPT** | ✅ 核心支持 | 对话交互，无文件系统（用对话记忆替代），无 subagent |
| **Gemini** | ✅ 核心支持 | 同 ChatGPT，对话交互为主 |
| **DeepSeek** | ✅ 核心支持 | 同 ChatGPT |
| **Kimi** | ✅ 核心支持 | 同 ChatGPT |

### 6.2 平台差异处理

| 能力 | Claude | ChatGPT/Gemini/DeepSeek/Kimi |
|------|--------|------------------------------|
| 交互模式 | AskUserQuestion 结构化选择 + 自由输入 | 自由对话为主，引导式提问 |
| 路由机制 | 菜单选项 + 自动检测 | 对话中自然引导用户选择模式 |
| 分析能力 | subagent 并行分析各章节 | 串行分析，建议分章节输入 |
| 文件持久化 | Read/Write Markdown 文件 | 对话记忆（用户可手动复制到笔记） |
| 引用验证 | 可选 WebSearch | 对话中提示用户自查 |
| 长文本处理 | 文件读取 + 分块分析 | 提示用户分章节提供 |

### 6.3 跨平台核心策略

**核心设计原则**：一份 Prompt 逻辑，适配所有平台

```
所有平台通用（Prompt 核心）：
├── 模式路由逻辑
├── 各模式的分析规则与输出格式
├── 学科结构模板
├── 逻辑谬误检测规则
└── 学术对话匹配规则

Claude Code 增强（可选）：
├── AskUserQuestion 交互
├── Agent subagent 并行分析
├── Read/Write 文件持久化
└── WebSearch 引用验证
```

对于非 Claude 平台，核心差别在于：
1. **路由方式**：用自然语言引导替代菜单选择（"您想做什么？A结构设计 B写作体验..."）
2. **持久化**：提示用户"建议将分析结果保存到笔记中"
3. **分析范围**：提示用户分章节输入而非一次性读入整篇文档
4. **并行性**：所有分析串行执行
